## 第一次用 Codex，不要先想着“让它全自动做项目”

一个普通人想做网站、改页面、修文章、加功能，最容易卡住的地方不是想法不够多，而是不知道怎么把想法变成项目里的一个具体改动。

你不是没有想法，而是经常卡在“我知道哪里不舒服，但不知道该让谁改、怎么改、改完怎么检查”。

你可能不是程序员，不熟悉文件结构，也不确定一个按钮、一个页面、一个样式到底藏在哪个文件里。看到 Codex 之后，很自然会冒出一个念头：能不能直接让它把网站做好？

真正危险的地方就在这里。

Codex 能执行很多事，但它不能代替你判断项目方向。它可以读文件、改代码、跑构建、查报错，也可以告诉你改了哪里。但一个页面该不该改，一个功能是不是符合用户，一个内容模块是不是应该保留，这些判断仍然要由人来做。

第一次用 Codex，真正要练的不是“全自动”，而是把一个具体问题交代清楚：

```text
把一个具体问题说清楚
把允许修改的范围说清楚
把不能动的内容说清楚
让它执行
再由人检查结果
```

这件事听起来不炫，但它很重要。普通人参与真实项目，不是靠一句“帮我优化一下”把项目交出去，而是把模糊想法拆成可以执行、可以检查、可以回退的小任务。

凡人修AI这个站点就是这样一点点做出来的。很多改动不是一次性重做，而是先发现一个具体问题，再把问题、范围、边界和验收标准写清楚，让 Codex 执行，最后由人看页面、看差异、看构建结果。

## Codex 在项目里到底负责什么

Codex 在项目里更像一个执行型协作伙伴。它擅长把已经说清楚的任务，落到文件、页面和命令上。

它可以帮你看项目结构，找到相关文件，修改代码和样式，调整页面结构，跑构建命令，检查报错，输出修改报告，也可以协助 Git 收口。

但它不应该负责项目定位、用户判断、商业承诺和最终验收。一个普通人用 Codex，最容易犯的错，是把“我不会代码”理解成“我不用判断”。恰恰相反，越是不写代码，越要把判断说清楚。

:::tutorial-role-split
title: Codex 能做什么 vs 人必须负责什么
you: 判断项目方向,说明用户场景,限定修改范围,检查体验结果,决定是否提交
codex: 阅读项目文件,查找相关模块,修改代码样式,执行构建检查,整理改动报告
:::

Codex 可以做执行，但不要让它替你决定方向。比如要不要删除某个模块，要不要改变首页定位，要不要改掉文章结构，这些不是纯技术问题，而是产品和内容判断。

如果你没有说清楚，它可能会为了完成任务改动太大；如果你没有限制范围，它可能会顺手改到别的页面；如果你只看构建通过，它可能已经改掉了你不想动的内容。

普通人用 Codex 的基本姿势，是先把任务变小，再把边界写清。

## 我现在更相信的一件事

我现在更相信的一件事是：Codex 的价值，不是让普通人不再判断，而是让普通人可以把判断变成可执行的项目改动。

过去很多普通人有想法，但离项目很远。你能感觉到页面不舒服，知道某个模块没必要，也能判断一段文案不合适，可你不知道怎么改文件、怎么跑构建、怎么确认有没有影响其他页面。

Codex 补上的，是中间这一段执行距离。

普通人用 Codex，最重要的不是“我不会代码也能全交给它”，而是“我可以把需求说清楚，再让它帮我落地”。

真正的协作关系更像这样：

```text
人判断方向
人限定边界
Codex 执行修改
构建负责验证
截图负责体验验收
最后人决定是否提交
```

这里面每一步都不能省。构建能告诉你页面有没有生成、命令有没有报错；截图能告诉你页面是否真的舒服；diff 能告诉你有没有改到不该改的地方；最后的提交决定，仍然应该由人来做。

## 一个真实例子：去掉文章页鸡肋的文章目录

凡人修AI的文章详情页，左侧原来有一个“文章目录”。它会自动列出文章里的标题，看起来很像文档站。

但凡人修AI不是文档站。它更像一条普通人的 AI 修炼路径。用户读一篇文章时，更需要沉浸正文，也更需要知道这篇文章的目标、建议节奏和所属路径。

文章目录在这个场景里有点鸡肋：目录项多，占注意力，分散阅读，还会让页面更像技术文档。

人先做判断：不要整个左侧栏都删，只去掉真正干扰阅读的文章目录。

原因很简单。文章目录解决的是“快速跳转”的问题，但凡人修AI的文章更需要用户顺着一条思路读下去。左侧真正有帮助的，不是十几个标题链接，而是提醒用户：这篇文章要完成什么、建议用什么节奏读、它属于哪条修炼路径。

接下来，人的任务不是自己去翻代码，而是把判断讲清楚：保留有用的修炼提醒，去掉干扰阅读的目录感。

:::tutorial-flow
title: 真实案例：删除文章目录
发现问题|文章目录让页面更像文档站，分散了连续阅读的注意力。
做出判断|保留有帮助的修炼提醒，去掉干扰阅读的目录感。
交给 Codex|把这个判断落到侧栏结构和页面交互里。
回到页面|观察左侧是否更安静，正文是否更适合沉浸阅读。
形成方法|先判断该留什么、该删什么，再让 Codex 去执行。
:::

Codex 负责把这个判断落到页面里。它去处理目录展示、侧栏结构和相关交互，让页面不再像文档站，而更像一篇可以沉浸阅读的修炼文章。

改完以后，人再回到页面上看：左侧是否还保留目标和路径提醒，正文阅读是否更安静，页面有没有变得更像凡人修AI，而不是普通技术文档。

结果是，文章目录被去掉了，左侧留下“本篇目标、建议节奏、所属路径”三块提醒，并在桌面端跟随正文滚动。这个改动本身不复杂，但它体现了普通人使用 Codex 的关键方法：先判断什么该留、什么该删，再让 Codex 去执行。

这个例子最重要的不是“Codex 会删目录”，而是：Codex 不是自己决定删目录，是人在判断后，把一个具体改动讲清楚。

## 普通人给 Codex 提需求，要说清 6 件事

如果你想让 Codex 帮你参与一个真实项目，不要只说“帮我优化一下”。这句话太大，边界也太模糊。

更稳的做法，是每次只处理一个具体问题，并说清 6 件事。

:::tutorial-task-card
title: 提需求要说清 6 件事
target: 把一个模糊项目想法变成 Codex 能执行的小任务。
includes: 项目目标|当前问题|希望结果|允许修改范围|禁止修改范围|验收方式
standard: Codex 知道要改哪里|知道哪些不能碰|知道完成后怎么检查
:::

第一，项目目标。你要说明这个网站、页面或功能是为了什么。目标不是越大越好，而是让 Codex 知道这次改动服务于什么方向。

第二，当前问题。哪个页面、哪个模块、哪里不好，要尽量具体。比如“文章页左侧目录太吵”，比“文章页不够高级”更容易执行。

第三，希望结果。你希望它变成什么样。是更克制、更清楚、更适合阅读，还是按钮跳转更明确？结果说得越清楚，执行越不容易跑偏。

第四，允许修改范围。哪些文件可以动，哪些模块可以调整。即使你不知道完整文件名，也可以说清页面范围和功能范围。

第五，禁止修改范围。哪些正文、数据、页面、功能不能碰。这一点很关键。真实项目里，保护边界和完成任务一样重要。

第六，验收方式。完成后执行什么命令，检查哪些页面，是否需要截图，是否要看搜索索引、sitemap、链接和移动端。

## 用 Codex 做项目时，最容易踩的坑

Codex 很强，但项目里真正容易出问题的地方，往往不是它不会执行，而是人没有把任务说清楚。

:::tutorial-compare-table
title: 常见踩坑提醒
需求太大|把任务缩成一个页面、一个模块、一个具体问题。
边界不清|提前写清允许修改和禁止修改的范围。
只看构建通过|构建之后还要看页面体验和移动端显示。
不看 diff 就提交|提交前要确认没有改到不该改的文件。
让 Codex 判断方向|方向由人判断，Codex 负责把判断落地。
:::

第一个坑，是需求太大。比如“帮我优化整个网站”。这种需求听起来明确，其实没法验收。优化哪里？首页、文章页、导航、内容地图，还是移动端？每个地方都可能变成一堆改动。

第二个坑，是边界不清。你没有说哪些正文不能动、哪些页面不能改、哪些功能必须保留，Codex 就只能根据自己的理解推进。真实项目里，最怕的不是没改，而是改多了。

第三个坑，是只看构建通过。构建通过只说明页面能生成，不代表体验好，也不代表定位正确。一个按钮位置、一个标题大小、一个左侧栏是否抢注意力，都要人来看。

第四个坑，是不看 diff 就提交。哪怕结果看起来没问题，也要看改了哪些文件。普通人不一定能读懂每一行代码，但至少要知道改动范围是不是符合自己允许的边界。

第五个坑，是让 Codex 判断方向。工具执行能力强，但方向应该由人决定。不要把本地路径、密钥、账号、隐私内容暴露给 Codex，也不要让它擅自修改大量文件。最后一定要由人验收。

## 直接复制这段，给 Codex 提一个清楚需求

下面这段可以直接复制。你不用一次填得很完美，但至少要把目标、问题、范围和检查方式写出来。

```text
请帮我处理项目中的一个具体问题。

项目目标是：
当前遇到的问题是：
问题出现在哪个页面或功能：
我希望最终效果是：
允许你修改的文件范围：
不要修改的内容：
需要特别保留的功能：
完成后请执行的检查命令：
需要重点检查的页面：
我希望你完成后输出：

1. 修改了哪些文件
2. 为什么这样修改
3. 是否影响其他页面
4. 是否修改了我禁止修改的内容
5. 构建是否通过
6. 是否建议人工截图验收

要求：

不要擅自扩大修改范围。
不要修改文章正文、标题、slug 或 frontmatter，除非我明确允许。
不要提交 git commit。
如果你发现风险，请先说明，不要直接改。
```

这段话的核心不是格式，而是提醒你：不要把项目交给一个模糊指令。你越能说清楚，Codex 越像协作伙伴；你越模糊，它越容易变成一个不可控的自动执行器。

## 后面继续怎么练

当你能把一个小问题说清楚，就可以继续学习凡人修AI的 Codex 协作流程。

Codex 专区后面的内容，不会从安装和参数讲起，而会继续从凡人修AI的真实改动里，拆解普通人怎么提需求、怎么验收、怎么把 AI 变成项目协作力。

先从小范围修改开始，逐步练会：提需求、限边界、看结果、跑构建、做验收，再决定是否提交。

普通人参与真实 AI 项目，不是一下子变成程序员，也不是把方向交给工具，而是学会把自己的判断变成可执行的小改动。
