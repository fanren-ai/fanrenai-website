# 高质量Prompt案例 文章设计稿

## 一、文章基本信息

1. 建议标题：高质量Prompt案例：不要只复制，先看懂它为什么有效
2. slug 建议：012-good-prompt-examples
3. 分类建议：Prompt与工作流
4. status 建议：published
5. description 建议：面向刚开始练习 Prompt 的普通人，通过客户跟进消息案例，讲清好 Prompt 为什么更稳定，以及如何把别人的案例改成自己的真实任务。
6. 预计正文长度：1800-2400 字

## 二、本篇如何避免和 011 同构

011《Prompt工作流入门》解决的是：一个任务不要一次性问完，而要拆成连续步骤。

012《高质量Prompt案例》解决的是：一条 Prompt 本身如何写得更清楚、更贴近场景，以及如何拆解别人的好案例。

011 的重点是任务推进顺序，让用户从“找万能提示词”转向“把任务拆成步骤”。

012 的重点是 Prompt 质量判断和案例改写，让用户从“复制别人的提示内容”转向“看懂结构，再换成自己的场景”。

因此，本篇不再沿用 011 的“提示内容囤积”切入，也不继续强调连续推进。开头直接放两种问法对比：同样是写客户跟进消息，一句泛泛要求和一条信息完整的 Prompt，为什么会得到完全不同的结果。

## 三、用户真实痛点

1. 复制别人的 Prompt 后，结果不适合自己的场景。
2. 看到别人写得很长，但不知道到底好在哪里。
3. 自己写 Prompt 时，总是只写一句模糊需求。
4. 不知道什么时候该补充身份、场景、目标、限制和格式。
5. 不知道如何要求 AI 给多个版本。
6. 不知道什么时候该让 AI 先问问题。
7. 不知道怎么把别人的案例改成自己的工作任务。
8. 不知道如何判断生成结果是否真的可用。

## 四、开头真实场景设计

开头不定义 Prompt，也不从工具入口或提示词收藏切入。

从“同一个任务，两种问法，结果差很多”开始：

同样是让 AI 写一段客户跟进消息。

第一种问法：

帮我写一段客户跟进话术。

第二种问法：

我是做企业服务销售的。这个客户上周了解过我们的方案，但还没有确认是否继续沟通。请帮我写一段微信跟进消息，语气自然、克制，不要催得太急，目标是约一个 15 分钟电话沟通。请给我 3 个版本，并说明每个版本适合什么场景。

两种问法都不复杂，但结果会明显不同。

第一种问法，AI 只能猜。它不知道你是谁，不知道客户现在处在哪个阶段，也不知道这段话是发在微信里，还是写在正式邮件里。

第二种问法，AI 至少知道：你是谁、客户状态、使用场景、语气限制、最终目标和输出方式。

这就是高质量 Prompt 的起点。

不是越长越好，而是该说的信息说清楚。

## 五、文章核心判断

### 我现在更相信的一件事

高质量 Prompt 不是一段很长、很复杂、看起来很专业的话。

更重要的是，它让 AI 明白五件事：

1. 你是谁。
2. 你要完成什么。
3. 背景是什么。
4. 有哪些限制。
5. 最终要输出成什么样。

普通人学 Prompt，不是为了背模板。

更有用的方式，是拆解一个好案例，看它哪些地方说清楚了，再把这些结构迁移到自己的任务里。

不要把 Prompt 当成万能句子。

更适合的方式，是把它当成一份清楚的任务说明。

## 六、差异化文章结构设计

1. 同一个任务，为什么两种问法结果差很多？
   - 用客户跟进消息开场。
   - 展示一句泛泛要求和一条信息完整 Prompt 的差别。

2. 好 Prompt 不是越长越好，而是信息够清楚
   - 说明长度不是标准。
   - 判断重点是身份、任务、背景、限制和输出格式是否明确。

3. 拆一个高质量 Prompt，看它到底好在哪里
   - 把客户跟进案例拆成 6 个片段。
   - 让读者看到每个片段在减少 AI 猜测。

4. 普通人写 Prompt，先补这 5 类信息
   - 身份、任务、背景、限制、输出格式。
   - 每一类都用日常工作语言解释。

5. 复制案例前，先改成自己的场景
   - 说明不能整段照搬。
   - 重点替换行业、对象、当前状态、语气和输出格式。

6. 我现在更相信的一件事
   - 放在案例拆解和迁移方法之后。
   - 收束为“看懂结构比背模板更重要”。

7. 今天把一句泛提示词改成高质量版本
   - 选一句常用泛提示词。
   - 按 5 类信息补完整。

8. 直接复制这段，帮你检查一条 Prompt
   - 提供完整检查提示内容和追问句。
   - 正文发布时使用复制容器。

9. 后面继续怎么练
   - 引导用户看到案例先拆结构，再改场景。

## 七、高质量 Prompt 的 5 类信息

这一节不写成技巧列表，而是写成普通人能照着补的信息。

1. 身份
   - 说明我是谁，或者希望 AI 扮演什么角色。
   - 示例：我是企业服务销售，你是一名擅长客户沟通的顾问。
   - 作用：让 AI 知道回答要站在哪个位置。

2. 任务
   - 说明这次到底要完成什么。
   - 示例：写一段客户跟进微信。
   - 作用：避免 AI 把任务扩展成无关方案。

3. 背景
   - 说明当前发生了什么，为什么要做这件事。
   - 示例：客户上周了解过方案，但还没有确认下一步。
   - 作用：让输出贴近真实关系和进度。

4. 限制
   - 说明哪些不能做，哪些语气不要出现。
   - 示例：不要催太急，不要夸大承诺，不要像广告。
   - 作用：减少 AI 常见的过度包装。

5. 输出格式
   - 说明最终希望 AI 怎么交付。
   - 示例：给 3 个版本，并说明每个版本适合什么场景。
   - 作用：让结果更容易比较和选择。

## 八、案例拆解设计

案例任务：写一段客户跟进消息。

模糊 Prompt：

帮我写一段客户跟进话术。

高质量 Prompt：

我是做企业服务销售的。这个客户上周了解过我们的方案，但还没有确认是否继续沟通。请帮我写一段微信跟进消息，语气自然、克制，不要催得太急，目标是约一个 15 分钟电话沟通。请给我 3 个版本，并说明每个版本适合什么场景。

拆解方式：

1. “我是做企业服务销售的”
   - 作用：说明身份和行业。
   - 可替换成：我是课程顾问、社群运营、老师、小店老板、自由职业者。

2. “客户上周了解过我们的方案”
   - 作用：说明客户当前阶段。
   - 可替换成：客户刚咨询、已经试用、还在比较、之前拒绝过。

3. “微信跟进消息”
   - 作用：说明使用场景。
   - 可替换成：邮件、社群私信、电话前短信、朋友圈私聊。

4. “语气自然、克制，不要催得太急”
   - 作用：说明语气限制。
   - 可替换成：正式、亲和、简洁、不要像广告、不要太强势。

5. “目标是约一个 15 分钟电话沟通”
   - 作用：说明真实目标。
   - 可替换成：确认需求、约时间、收集反馈、提醒提交资料。

6. “给我 3 个版本，并说明每个版本适合什么场景”
   - 作用：说明输出格式。
   - 可替换成：表格、清单、三个版本、优先级排序、步骤说明。

## 九、判断提醒设计

文章中要自然提醒：好 Prompt 可以提高结果稳定性，但不能保证结果一定正确、一定适合发布、一定适合真实关系。

用户仍然需要判断：

1. 背景是否真实。
2. 语气是否适合对象。
3. 是否有过度承诺。
4. 是否遗漏关键信息。
5. 是否适合直接发送或发布。

通俗解释：

AI 可以帮你生成更贴近场景的内容，但不能替你承担结果。

如果是发给客户、领导、公开平台，或者涉及商业承诺、财务、法律、医疗、隐私内容，最好再自己检查一遍。

## 十、今天的小任务设计

任务：把一句泛提示词改成高质量版本。

可以从这些句子里选一条：

1. 帮我写一段客户跟进话术。
2. 帮我写一篇小红书文案。
3. 帮我做一份学习计划。
4. 帮我总结这篇文章。
5. 帮我写一份周报。

任务目标不是马上得到最终内容，而是学会把泛提示词补充成更清楚的任务说明。

操作顺序：

1. 选一句自己常用的泛提示词。
2. 补充身份。
3. 补充任务目标。
4. 补充背景。
5. 补充限制。
6. 规定输出格式。
7. 让 AI 检查这条 Prompt 还缺什么。

任务后的判断：

如果用户能把一句泛提示词改成包含身份、任务、背景、限制和输出格式的版本，并知道哪些信息还需要补充，这篇文章就达到目的。

## 十一、可复制内容设计

标题建议：直接复制这段，帮你检查一条 Prompt

提示内容：

```text
我写了一条 Prompt，但不确定它够不够清楚。

我的 Prompt 是：

请你帮我检查这条 Prompt 是否完整，并按下面结构分析：

1. 任务是否清楚
2. 是否说明了使用场景
3. 是否补充了必要背景
4. 是否说明了限制条件
5. 是否规定了输出格式
6. 是否还缺少关键信息

请你再帮我做两件事：

1. 先指出这条 Prompt 的问题
2. 再帮我改成一个更清楚、更适合真实使用的版本

要求：

1. 不要写得太复杂
2. 不要堆专业术语
3. 保留我的原始目的
4. 如果信息不足，请先问我 3 个问题
5. 输出格式用表格
```

追问句：

```text
请基于我的具体场景，再帮我改成一个可以直接使用的版本。
```

正文发布时，这段内容要放进复制容器，右上角有“复制”按钮，点击后复制完整文本。

## 十二、CSS 视觉模块设计

本篇不使用插图，只使用 CSS 视觉模块。视觉模块服务案例拆解和 Prompt 改写，不做装饰。

1. 模糊 Prompt vs 高质量 Prompt
   - 左侧：帮我写一段客户跟进话术。
   - 右侧：说明身份、客户状态、使用场景、语气限制、目标和输出格式。
   - 作用：让用户直观看到两种问法差在哪里。

2. 高质量 Prompt 5 类信息
   - 内容：身份、任务、背景、限制、输出格式。
   - 作用：给用户一个检查 Prompt 的基础结构。

3. 案例拆解表
   - 内容：原句片段、起到什么作用、可以怎么替换成自己的场景。
   - 作用：让用户知道好案例不是用来照搬，而是用来拆结构。

4. 复制前先改这 5 处
   - 内容：行业、对象、当前状态、语气、输出格式。
   - 作用：降低直接照搬导致不适配的概率。

5. Prompt 结果判断卡
   - 内容：背景真实、语气适合、不过度承诺、信息完整、可发送。
   - 作用：提醒用户不要只看生成结果是否流畅。

## 十三、结尾设计

结尾不要写成标准化归纳，也不要把高质量 Prompt 包装成神奇技巧。

结尾要让读者感觉：

我不是又收藏了一条 Prompt，而是知道了一条好 Prompt 为什么好。

结尾方向：

如果你今天真的把一句泛提示词改成了更清楚的版本，就已经完成了一次有效练习。

以后看到别人的 Prompt，不要急着整段复制。

先看它有没有说清身份、任务、背景、限制和输出格式。

再把这些部分换成自己的真实场景。

这样你不是在背模板，而是在学会写自己的任务说明。
